ITOps v2 — Setup & Deployment Guide

Version 2.0.0-alpha.1 · This guide is self-contained: a deployment at a client site should succeed using only this document.

1. Overview

ITOps v2 is a self-hosted IT asset management portal (assets, checkout/checkin, printers, licenses & contracts, inventory) with multi-currency support, configurable role permissions, and three authentication options (OIDC, LDAP/LDAPS, local accounts). It ships as a Docker Compose stack: the application container and a dedicated PostgreSQL 16 container.

3. Installation

1Create the data directories:

mkdir -p /opt/appdata/itops2/{pgdata,attachments,certs}

2Place docker-compose.yml in /opt/appdata/itops2/ (or clone the repo folder) and copy .env.example to .env.

3Edit .env:

Do not put inline comments on value lines. Set SECRET_KEY and DB_PASSWORD to long random strings, and write the DB password literally into DATABASE_URL as well — variable interpolation does not happen inside env files.

4Start the stack:

docker compose up -d

Database migrations run automatically on startup. Verify with:

curl http://localhost:8004/healthz

4. First login (break-glass admin)

On first boot the application creates a local administrator from BREAKGLASS_USERNAME / BREAKGLASS_PASSWORD.

This account is a permanent break-glass account: it always authenticates locally, even when OIDC or LDAP is enabled, and cannot be deactivated. Change its password immediately after first login.

5. Reverse proxy & TLS (expanded in Phase 2+)

Point your reverse proxy at port 8004. For internal PKI (e.g. Step-CA), mount the root certificate into /opt/appdata/itops2/certs/ and set CA_CERT_PATH so outbound HTTPS calls (OIDC provider) are trusted.

6. Authentication providers (LDAP: Phase 3, remaining)

6.1 OIDC (e.g. Authentik)

Go to Settings → Authentication, enable OIDC, and fill in:

FieldNotes
Issuer URLThe provider's base issuer, e.g. https://auth.home.internal/application/o/itops2/. The app appends /.well-known/openid-configuration to this value to fetch the discovery document on every login — enter the issuer exactly as the provider documents it (including any trailing slash the provider expects), not the discovery URL itself.
Client ID / Client secretFrom the provider's application/client config.
Group → role map (JSON)e.g. {"itops-admins": "admin", "it-staff": "technician"}. Highest-privilege matched group wins (admin > manager > technician > viewer).
Default roleRole for users in no mapped group; leave empty to deny login.
Redirect URI registration: the exact URI to register in the provider's application config is shown live on the Settings → Authentication page (Redirect URI to register with the provider: ...), and resolves to https://<your-domain>/auth/oidc/callback. Register this before enabling OIDC — a mismatch is the most common first-login failure. If the app sits behind a reverse proxy, uvicorn must run with --proxy-headers (already configured) so this URL is generated with the correct public scheme/host, not the internal container address.

Scopes requested: openid profile email. The provider must expose a standard userinfo_endpoint returning preferred_username (or email as a fallback), email, name, and a groups claim for the role mapping to work.

6.2 LDAP / LDAPS (not yet implemented)

Planned: bind + create-on-first-login, group→role mapping via an auth.ldap.group_role_map JSON setting. Settings keys exist as placeholders (auth.ldap.url, bind_dn, bind_password, user_base, user_filter, group_role_map) but there is no bind/search logic, login route, or Settings UI for them yet.

7. Permissions (Phase 3)

Four fixed roles (Admin, Manager, Technician, Viewer) with a configurable permission grid — Settings → Permissions.

8. Catalog: core lookups

Sidebar → Catalog. Feeds Assets: every asset references a model, which in turn references a manufacturer and a category.

PageNotes
Manufacturers, Categories, LocationsSimple name-only lookups. Delete is blocked (with an explanatory message) while a row is still referenced elsewhere.
ModelsBelongs to one manufacturer + one category. Optional Depreciation (months) and EOL (months) override the global defaults set in Settings → General for that model only — leave blank to inherit.
Status LabelsA display name plus a fixed type (deployable / deployed / pending / archived). Asset workflow rules key off the type, not the label text — you can rename a label freely without breaking workflow rules, as long as its type stays put.
CompaniesOnly visible/editable with the companies.manage permission (Admin by default) — unlike the other Catalog pages, there is no separate view-only permission for Companies.

Permissions: catalog.view (read-only) / catalog.manage (create, rename, delete) cover Manufacturers, Categories, Locations, Models, and Status Labels. Companies uses companies.manage alone for both.

9. Currency & localisation

Sidebar → Settings → Currency.

ConceptNotes
CurrenciesISO 4217 code + display symbol + active toggle. Seeded on first boot with SCR, USD, GBP, EUR (edits to symbol/active are never overwritten by the seeder on later restarts). The currency selected as Settings → General → Default currency cannot be deleted — change the default first.
Exchange ratesManual entry only — no external rate API. Each row is a dated rate: 1 from currency = rate × to currency, effective from a given date, so historical asset values can be computed at their purchase-date rate rather than today's rate. At most one rate per (from, to, date) triple.

UK date formats, English only, throughout the app.

10. Assets, checkout/checkin & attachments

Sidebar → Assets. The core inventory record — tag, serial, model (→ manufacturer/category), status, company, location, purchase date/cost/currency, warranty, and depreciation/EOL overrides.

The filter bar above the list (status label, category, model, location, company when multi-company is on, checked-out state, and a free-text search over tag/serial/model) narrows the table instantly as you change a field — no page reload — and the current filters are reflected in the URL, so a filtered view can be bookmarked or shared.

10.1 Asset tags

Leave the tag field blank on create to auto-generate one: the app scans existing tags matching Settings → General → Asset tag prefix, takes the highest numeric suffix + 1, and zero-pads it to Asset tag padding digits. Tags are always manually editable afterward, including on existing assets — duplicates are rejected with a clear message, not a server error.

10.2 Status lifecycle

Every status label has a fixed type (§8) that drives what you can do with an asset — the label's display name is cosmetic, the type is what matters:
  • deployable — the only status an asset can be checked out from.
  • deployed — reachable only via checkout, never through a direct status edit. This is enforced server-side even if you try to force it.
  • pending — not checkoutable; a normal status edit moves it elsewhere when it's ready (e.g. back to a deployable label after repair).
  • archived — the "soft delete" state. Archived assets are read-only except for a Restore action, which requires picking a destination status (there's no one-click auto-restore).
An asset that's currently checked out (deployed) can't have its status changed directly — checkin first.

10.3 Checkout / checkin

On an asset's detail page, checkout requires picking exactly one target — a user, a location, or (for accessories like docking stations) another asset — plus a destination status restricted to deployed-type labels. An optional expected checkin date can be set for due-date tracking. Checkin accepts any non-deployed destination status (e.g. back to deployable, or to pending if it came back broken) and closes out the checkout in the asset's history table. The app enforces at most one open checkout per asset at the database level, not just in the UI.

10.4 Attachments

Files (manuals, photos, purchase receipts) attach to an asset from its detail page — max 25 MB per file. Stored on the /data/attachments volume (host: /opt/appdata/itops2/attachments), organized as attachments/<entity>/<id>/<file>, e.g. attachments/asset/42/… (maintenance records use the same mechanism under attachments/maintenance/<id>/… — see §11.1). Uploads and deletes are blocked while an asset is archived.

10.5 Deleting assets

An asset can only be permanently deleted if it has zero checkout history, zero attachments, and zero maintenance records — all three are exactly the things that make an asset's audit trail worth keeping. Any other asset can only be archived (§10.2), never hard-deleted, so history is never silently lost.

11. Maintenance records & Printers

11.1 Maintenance records

A "Maintenance" section on every asset's detail page (not printer-specific) logs repairs, routine maintenance, and upgrades: date, type, description, an optional cost (if given, a currency is required — used for the Printers cost totals below), and who performed the work as free text (covers external vendors, who aren't user accounts in this system). Files (receipts, photos) attach per record the same way asset-level attachments work.

11.2 Printers

Sidebar → Printers — not a separate entity, a filtered view over assets whose model's category is named "Printer" (§8, case insensitive). The list shows IP address, status, location, and each printer's total maintenance cost, converted to the default currency using the dated exchange-rate table (§9) at each record's own date — a record in a currency with no applicable rate is excluded from the total and flagged with a warning icon rather than silently shown wrong.

IP address, hostname, and consumable notes are set from the printer asset's own detail page (a "Printer Details" section appears automatically once its category is "Printer"), editable by anyone with the printers.manage permission.

The list has its own filter bar — location, status, and a free-text search over hostname/IP address.

12. Licenses & Contracts

Sidebar → Licenses & Contracts. One unified simple module for software licenses, support contracts, and subscriptions — not Snipe-IT-style per-seat license tracking.

FieldNotes
Typelicense / contract / subscription.
VendorFree text — not tied to a user account.
End / renewal dateRequired. The list sorts by this and highlights rows red (expired) or yellow (expiring soon) based on Settings → General → Contract renewal alert lead time.
CostRequires a currency to be picked if set, same rule as everywhere else in the app.
Covered assetsOptional, many-to-many — a support contract can cover several assets. Linking/unlinking never blocks deleting either side.

Attachments (agreements, invoices) work the same as on assets, from the contract's own detail page.

The list's filter bar covers type, renewal state (expiring soon / expired / active), and a free-text search over vendor.

13. Inventory

Sidebar → Inventory. Quantity-tracked consumables and spares, sharing the same category list as Assets (Catalog → Categories).

Quantity only ever changes via the Adjust action — a +/- amount plus a required reason (e.g. "received shipment", "used on IT-0012"). There is no way to type a new quantity directly; every change is logged with who, when, the delta, and the reason. An adjustment that would take stock negative is rejected, not clamped to zero.

Setting a minimum quantity on an item flags it (a yellow row + "low" badge) once quantity drops to or below that threshold.

The list's filter bar covers category, location, a "low stock only" toggle, and a free-text search over item name.

14. Notifications (SMTP)

Settings → Notifications. Configure once per install; nothing is sent until Enable email notifications is switched on.

FieldNotes
SMTP host / port25 for an unauthenticated internal relay, 587 (STARTTLS) or 465 (implicit TLS) for an authenticated provider (O365, Gmail, etc.).
SecurityMust match the port — None for port 25, STARTTLS for port 587 (plaintext connect, then upgrade), TLS for port 465 (encrypted from the first byte). These are two different wire protocols; picking the wrong one for a given port fails at the TLS handshake with a version-mismatch error, not a helpful auth error.
Username / passwordLeave both blank for an unauthenticated relay.
From addressSender shown in the notification emails.
Leaving username/password blank does the right thing automatically — the app passes them to the mail library as absent, not as empty strings, which matters for relays that reject an AUTH attempt on an unauthenticated port.

Use the Send test button at the bottom of the page to confirm the relay works before relying on it — it reports the real failure (connection refused, auth rejected, etc.) rather than silently doing nothing.

14.1 Events & per-user subscriptions

EventSent to
Asset checked outThe checkout's target user (if it has an email) always; subscribed users with Assets view access on request.
Asset checked inSame as above, for the asset's previous holder.
Warranty expiring soonSubscribed users with Assets view access — lead time from Settings → General → Warranty alert lead time.
Contract renewal dueSubscribed users with Licenses & Contracts view access — lead time from Contract renewal alert lead time.
Inventory low stockSubscribed users with Inventory view access.

Each user manages their own subscriptions on their Profile page (sidebar → username). Only events a user's role can actually receive are shown there — a role change immediately adds or removes the relevant checkboxes.

Warranty/renewal/low-stock are checked once a day automatically (no configuration needed) and batched into a single digest email per event type, so subscribers get one email listing everything due rather than one per item.

15. Dashboard

The landing page. Each card links straight into the relevant list, pre-filtered: asset counts by status, checkouts overdue / due back within 7 days, warranties expiring soon, contracts renewing soon, and low-stock inventory items. Only the cards for sections the signed-in user actually has view access to are shown.

When Settings → General → Scope users to their company is on, a user with a company assigned sees dashboard counts for their own company only. The Assets/Licenses & Contracts list pages themselves don't enforce this scoping yet — only the dashboard does, for now.
